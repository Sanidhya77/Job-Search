# Step 2 — Implementation Progress (08.05)

## 1. Updated System Description

The system described in Step 1 has moved from a written plan to a partial codebase, and this section reports on what is actually built now versus what is still ahead. The goal has not changed: the agent reads the user's CV and a short brief, finds matching jobs, scores each one against the CV, and produces a tailored CV, a cover letter, a job summary, and the apply link for every match above the relevance threshold.

The deterministic half of the system is now implemented and tested. This includes the four domain dataclasses (`UserProfile`, `Job`, `ScoredJob`, `Application`) in `src/job_agent/models.py`, the configuration loader in `src/job_agent/config.py` that reads API keys and settings from a local .env file at startup, the CV reader tool in `src/job_agent/tools/cv_reader.py` that handles both PDF and DOCX input, and the output writer in `src/job_agent/tools/output_writer.py` that writes the per-job output folder with sanitised folder names. The test suite covers all of this with 32 passing tests across the modules, including edge cases for malformed inputs and folder name collisions.

The LLM-based half of the system is the next milestone. This includes the OpenAI client wrapper, the CV analyser that turns plain CV text into a structured `UserProfile`, the scorer that rates each candidate job against the profile, the rewriter that produces tailored CV and cover letter text, the SerpApi job search tool, and the agent orchestrator that wires everything together. Empty placeholder files for `agent.py`, `cli.py`, and `tools/job_search.py` are already in the repository, with the actual implementation scheduled across the remaining days.

A few things changed from the Step 1 plan because of what I learned during implementation. The DOCX path in the CV reader walks tables explicitly because `python-docx` excludes table cell text from `document.paragraphs`, which would silently drop the skills section in CV templates that use a sidebar table. The job-title-to-folder-name sanitisation grew from a one-line string replace in my head to a six-step pipeline with eight parametrised test cases, because real job titles contain unicode (Müller), gender markers like (m/f/d), emoji, slashes, and at-signs, all of which break the file system in different ways. And the original Step 1 list did not explicitly mention dataclasses or `pytest.parametrize`, both of which turned out to shape the codebase more than expected.

## 2. Refined List of Programming Concepts Actually Used

The Step 1 list of concepts has been re-evaluated against the code that actually exists. Each concept below is tagged with its current status: used as expected, used differently than expected, not yet used because the relevant module is not built, or added during implementation and not in the original Step 1 list.

- **Configuration management — used as expected.** Implemented in `src/job_agent/config.py` as a frozen dataclass loaded from .env via python-dotenv at startup. The loader validates that required keys are present and that the relevance threshold is an integer between 0 and 100, raising a `ValueError` with a clear message at startup rather than failing later inside an API call.
- **File I/O across multiple formats — used as expected.** Implemented in `src/job_agent/tools/cv_reader.py` for reading (pdfplumber for PDF, python-docx for DOCX) and `src/job_agent/tools/output_writer.py` for writing (python-docx for DOCX, `pathlib.Path.write_text` for plain text files).
- **Modular design — used as expected.** Each tool sits in its own module under `src/job_agent/tools/` with a single public function and a single custom exception type. Tests for each module run independently.
- **Error handling and graceful degradation — used as expected.** Each tool defines its own exception type (`CVReaderError`, `OutputWriterError`) that wraps any underlying parser failure. Folder name collisions are handled by appending a numeric suffix rather than crashing or overwriting.
- **String processing and sanitisation — used and turned out more important than expected.** The sanitisation pipeline in `output_writer.py` applies NFKD unicode normalisation, regex-based gender marker stripping, non-alphanumeric replacement with underscores, run collapsing, truncation, and an `Untitled_Job` fallback. Eight parametrised test cases cover the input patterns observed in real job titles. This is now the most thoroughly tested pure-Python function in the project.
- **API integration and authentication — partially used.** API key loading and validation are done; the actual HTTP calls (SerpApi via `google-search-results`, OpenAI via `openai`) are not yet wired up because the corresponding modules are next on the schedule.
- **Tool-calling and agent orchestration — not yet used.** The orchestrator in `agent.py` is currently a placeholder. It is scheduled for implementation alongside the LLM modules.
- **Structured data extraction — not yet used.** The CV analyser that turns raw CV text into a `UserProfile` requires the LLM client wrapper, which is the next module to build.
- **Ranking and filtering with LLM scoring — not yet used.** Same dependency: requires the LLM client and the CV analyser to be in place first.
- **Testing with mocks — partially used.** Fixture files for the CV reader exist (a sample PDF and DOCX generated by `tests/fixtures/generate_fixtures.py`), and tests for `output_writer.py` use `pytest`'s `tmp_path` fixture. Mocked responses for SerpApi and OpenAI will be added when those modules are built.
- **Version control hygiene — used as expected.** The repository has 11 meaningful commits at the time of this submission, with messages that describe what each commit added, plus a `.gitignore` that excludes the .env file, output folders, and user CVs.
- **Dataclasses and type-driven design — added during implementation, not in Step 1.** The four domain models in `src/job_agent/models.py` are dataclasses, and `Config` is a frozen dataclass. This pattern shapes every other module's signatures, because each tool either accepts or returns one of these typed objects rather than raw dicts.
- **`pytest.parametrize` for test compaction — added during implementation, not in Step 1.** The eight sanitisation test cases live in a single parametrised function instead of eight separate test functions, which makes the test file shorter and the test scenarios easier to read as a coherent set.

## 3. How These Concepts Are Applied in the Project

This section gives concrete code references for the concepts that are already implemented, so the grader can see where each one lives in the codebase rather than treating them as abstract theory.

The **configuration management** concept appears in `src/job_agent/config.py`. The `Config` dataclass is declared with `frozen=True`, which prevents any module from mutating settings after the loader has read them at startup. The `load_config()` function pulls each value from the environment, validates the required keys, and converts the relevance threshold to an integer with range checking. Tests in `tests/test_config.py` cover the happy path plus three failure modes (missing OPENAI_API_KEY, non-integer threshold, out-of-range threshold).

The **file I/O across formats** concept is split between `cv_reader.py` (input) and `output_writer.py` (output). The CV reader dispatches on file extension because the user always knows what file they uploaded, so content sniffing would solve a problem nobody has. The DOCX path walks both `document.paragraphs` and `document.tables` because skills sections in real CV templates often live in sidebar tables that python-docx excludes from the paragraph list. The reader returns the same kind of normalised string regardless of input format, so the future CV analyser does not need to know which format the user provided.

The **modular design** concept is visible in the directory layout itself. The `tools/` folder holds deterministic external integrations and the `llm/` folder (still empty) is reserved for LLM reasoning modules. Each tool exposes a single public function plus private helpers prefixed with an underscore. This separation makes it possible to test each tool in isolation, because mocking one tool does not leak into another.

The **error handling** concept is applied through tool-specific exception types. `CVReaderError` wraps any underlying parser failure (missing file, unsupported extension, corrupted PDF) so callers handle a single exception type instead of branching on whatever pdfplumber or python-docx happened to raise. `OutputWriterError` does the same on the write side. This means the agent's main loop, when it is built, will need only a small number of `try`/`except` blocks rather than dozens.

The **string sanitisation** concept is the largest single piece of pure-Python logic in the project so far. The `sanitize_job_title()` function in `output_writer.py` runs through six steps: NFKD unicode normalisation to fold "Müller" to "Muller", regex stripping of gender markers like `(m/f/d)` and `m/w/d`, replacement of all non-alphanumeric characters with underscores, collapsing of underscore runs, truncation to 80 characters to stay safe on Windows path limits, and a fallback to `Untitled_Job` when input sanitises to nothing. Each step exists because of an actual input pattern observed in real listings, not as defensive paranoia.

The **dataclasses pattern** is the thread that ties the whole project together, even though it was not in the Step 1 concept list. Every contract between modules is a dataclass with typed fields. The CV analyser will return a `UserProfile`. The job search will return a list of `Job`. The scorer will return `ScoredJob`. The rewriter will return an `Application`. This means each module's tests can build typed inputs directly and assert against typed outputs, instead of constructing fragile dictionary literals.

## 4. How Tools Are Integrated Into the System

Three of the four planned tools are integrated and tested. The fourth is scheduled for tomorrow.

**CV reader (`src/job_agent/tools/cv_reader.py`).** Takes a `pathlib.Path` pointing to a PDF or DOCX file and returns a normalised plain-text string. The agent will call this once at the start of each run, before any LLM step, with the path the user provided. The tool dispatches internally based on file extension and returns the same kind of string regardless of input format. Failures from either underlying parser are wrapped in a single `CVReaderError` with a message that distinguishes between a missing file, an unsupported extension, and a corrupted file, so the agent's main loop only has to handle one exception type per tool.

**Output writer (`src/job_agent/tools/output_writer.py`).** Takes a finished `Application` dataclass produced by the rewriter step and writes a per-job folder under the user's configured output directory. The folder contains four files: `custom_resume.docx` and `cover_letter.docx` written by python-docx, plus `job_summary.txt` and `apply_here.txt` written via `pathlib.Path.write_text`. The folder name is derived from the job title and company through the sanitisation pipeline described in section 3, with collision handling that appends a numeric suffix when the same role is run twice. The writer wraps any underlying failure in a custom `OutputWriterError`.

**SerpApi job search (`src/job_agent/tools/job_search.py`, scheduled).** This module is currently an empty placeholder. When implemented, it will accept a query string built from the user's brief, call SerpApi via the `google-search-results` package, parse the JSON response into a list of `Job` dataclasses, and return that list to the agent. Errors will be wrapped in a `JobSearchError`. This is Day 5 work, scheduled for tomorrow, with the first real API call captured as a fixture file so subsequent tests run offline.

**OpenAI integration (planned in `src/job_agent/llm/`, scheduled).** The LLM is not technically a tool in the same sense as the three above, because it is the agent's reasoning core rather than an external utility the agent calls for a single deterministic operation. It will be accessed through a thin client wrapper in `src/job_agent/llm/client.py` that handles retries on transient errors. The CV analyser, scorer, and rewriter modules will each call this wrapper with their own system prompts (kept as constants in `prompts.py` so they are version-controlled and reviewable). Real OpenAI responses captured during prompt iteration will be saved as fixture files for testing.
